## Analysis

### Instructions that succeed (offsets 0, 4, 8, 12, 16, 20, 24, 32, 40, 44, 48)

- `ldr x23, #+52` (offset 0): PC-relative literal load. Works because
  PCC includes `PERMIT_LOAD` and the load target is on the same page.
- `orr x26, xzr, x0` (offset 4): A64 ALU register. No memory access.
- `b +N` (offsets 8, 12, 40): A64 branch (no link). No memory access.
- `movz x24, #0x100` (offset 16): A64 ALU immediate. No memory access.
- `sub csp, csp, #16` (offset 20): Morello capability stack alloc. No
  memory access — only updates CSP register.
- `movz x5, #42` (offset 24): A64 ALU immediate. No memory access.
- `movz x5, #0` (offset 32): A64 ALU immediate. No memory access.
- `orr x0, xzr, x5` (offset 44): A64 ALU register. No memory access.
- `ret c30` (offset 48): Morello capability return. No memory access.

### Instructions that crash with SIGPROT (offsets 28, 36)

- `stur x5, [x23, #-8]` (offset 28): A64 data store through X23 (R10's
  mapped register) to the decoupled eBPF stack page. Crashes because
  QEMU Morello rejects stores from PROT_EXEC pages.
- `ldur x5, [x23, #-8]` (offset 36): A64 data load through X23.
  Crashes for the same reason.

### Literal pool (offset 52)

The 8-byte placeholder `0x0000000000000000` at offset 52 is patched by
`ubpf_compile_ex` after `mmap`'ing the separate eBPF stack page. The
patched value is `(uint64_t)stack_page + UBPF_EBPF_STACK_SIZE` (the
stack top, since eBPF grows downward).

### The M3 decoupled stack workaround

The prologue at offset 0 (`ldr x23, #+52`) loads the eBPF stack top
into X23 (the mapped register for eBPF R10). Subsequent eBPF STX/LDX
instructions use X23 as the base, going through DDC instead of CSP.

This design is architecturally correct for a system where W^X only
restricts writes TO executable pages. However, QEMU Morello rejects
ALL memory access FROM executable pages, making this workaround
insufficient on the current QEMU.
