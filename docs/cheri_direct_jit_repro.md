# Direct CHERI mmap JIT Repro - 2026-07-21

This note records the direct anonymous-`mmap` CHERI JIT problem, the root cause, and the fix. The repro is `workspace/test_cheri_direct_jit_repro.c` and runs with:

```sh
sg docker -c 'make run-cheri-direct-jit-repro'
```

It now passes.

## Root Cause

The anonymous executable mapping itself was not the blocker. The generated code was being entered without Morello C64/purecap execution state.

The key clue was that normal purecap ELF function pointers have an odd address, for example `0x112df1`, while our generated mmap entry originally had an even address like `0x408df000`. CheriBSD source confirms why that matters: `sys/arm64/include/cheric.h::trapframe_set_elr()` sets `PSR_C64` only when the target address has bit 0 set, then clears that bit before writing ELR.

So the generated bytes were valid, but the call entered them as non-C64 code. In that state, the instruction word that should behave as a capability-relative `ldr x0, [c0, #8]` faulted as a tag violation because the default data capability is untagged in this purecap process.

The fix is to form the callable JIT entry as a capmode address before sealing/calling it:

```c
void *entry = (void *)((uintptr_t)code | 1U);
entry = cheri_sentry_create(entry);
```

The uBPF integration now does the same for purecap mmap JIT code and keeps the raw mapping base separately for `munmap`, `mprotect`, and `ubpf_copy_jit()`.

## Evidence

The repro emits compiler-verified Morello instruction bytes into anonymous executable memory:

```text
mov w0, #42                 -> 52800540
gctag x8, c0                -> c2c09008
gclen x0, c0                -> c2c03000
ldr x0, [c0, #offset]       -> f9400000 | ((offset / 8) << 10)
ret c30                     -> c2c253c0
```

Before the fix:

- `mov`, `gctag`, and `gclen` worked from anonymous mmap code.
- `gctag c0` returned `1`, proving the argument capability arrived tagged.
- `gclen c0` returned `0x10`, proving the argument bounds were present.
- `ldr x0, [c0, #8]` still trapped with `PROT_CHERI_TAG`.
- Kernel logging showed `esr=0x92000028`, where the low DFSC `0x28` maps to CHERI tag violation.

CheriBSD source also clarified that `si_addr` for these CHERI aborts is the faulting ELR/instruction capability, not the data address. With `machdep.log_user_cheri_exceptions=1`, the kernel printed `far` as the actual data address, for example `ctx + 8` for the in-bounds load.

After the fix:

- in-bounds anonymous mmap generated load returns `0xfeedfacecafebeef`;
- out-of-bounds anonymous mmap generated load traps with `PROT_CHERI_BOUNDS`;
- raw unsealed capmode entry and sealed sentry capmode entry both work;
- scalar-address negative controls still trap with `PROT_CHERI_TAG`.

Representative passing output:

```text
entry capmode          tag=1 sealed=0 sentry=0 addr=0x408df001 base=0x408df000 len=0x1000
entry after mprot      tag=1 sealed=1 sentry=1 addr=0x408df001 base=0x408df000 len=0x1000
returned              0xfeedfacecafebeef
SIGPROT detail        si_code=1(PROT_CHERI_BOUNDS) si_addr=0x408df000
trap mc_spsr          0x64000200
trap c0               tag=1 addr=<ctx> base=<ctx> len=0x10
```

The changed `mc_spsr` value is important: `0x64000200` includes the C64 state bit, unlike the earlier failing `0x60000200` runs.

## uBPF Integration State

The direct CHERI mmap JIT is now the default purecap path for the narrow supported backend subset. The older shared-object object-JIT path is retained only as an opt-in fallback through:

```sh
UBPF_CHERI_USE_OBJJIT=1
```

Implemented in the uBPF submodule:

- purecap mmap JIT reserves `RWX+CAP` maximum permission, copies code, then narrows the mapping to `RX+CAP`;
- the VM stores both the callable capmode sentry and the raw mapping base;
- `munmap`, `mprotect`, and `ubpf_copy_jit()` use the raw mapping base;
- capability register aliases use `mov Cd, Cn`, not scalar `orr`, so `MOV64_REG rN, r1` preserves the context capability tag;
- the restricted CHERI backend maps BPF registers/temps to caller-saved registers because its current prologue intentionally avoids stack stores;
- unsupported memory operations fail closed at compile time for now.

Current passing checks:

```sh
sg docker -c 'make run-cheri-direct-jit-repro'
sg docker -c 'make run-cheri-objjit-compile'
sg docker -c 'make run-exploit-tests'
```

Despite the old target name, `run-cheri-objjit-compile` now exercises the default direct mmap JIT unless `UBPF_CHERI_USE_OBJJIT=1` is set.

## Remaining Caveats

This is not a complete Morello uBPF backend yet. The current backend deliberately supports a narrow claim:

- context capability loads from `r1` and simple aliases of `r1`;
- CHERI bounds trapping for out-of-bounds context loads;
- compile-time rejection for unsupported memory operations and unsafe context clobbering.

The next real engineering step is to replace the minimal no-store prologue with a correct purecap ABI prologue/epilogue, then expand memory support one instruction class at a time.
